[158][Heading 2] System Architecture
[159][Normal] This section presents the overall architecture of the system developed in the project. The platform was designed as a full-stack web system composed of two main applications: the CMS Admin application and the Frontweb application. These two applications were maintained in separate repositories and served different types of users. The CMS Admin application was used by internal users such as administrators and editors to manage content, settings, media, and other website data. [165][Normal] In addition to the overall system architecture, the project can also be viewed from a component-level development scope. Figure 2 shows the main application components that were directly related to development tasks during my internship.
[166][Normal] The system was deployed on Google Cloud. Both the CMS Admin application and the Frontweb application were hosted on Cloud Run [9]. User requests were routed through Cloud DNS and an external load balancer. Based on the route, requests could be directed either to the Frontweb application or to the CMS Admin application. For example, general public routes were handled by the Frontweb application, while CMS-related routes were handled by the CMS Admin application.
[167][Normal] The CMS Admin application was built using Payload CMS. It provided both the administrative interface and the CMS API. In the system architecture, the CMS Admin and CMS API belonged to the same Payload-based application, but they were accessed through different routes. The CMS interface was used by administrators and editors to manage content, while the CMS API provided endpoints for both CMS-side operations and Frontweb data consumption. For CMS-side operations, the API was used to manage or retrieve data for the administrative interface. For the Frontweb application, separate API endpoints were provided to expose published or public-facing data.
[168][Normal] PostgreSQL, hosted on Cloud SQL, was used as the main relational database of the system [10]. It stored CMS-related data such as articles, website settings, banner information, media metadata, and CMS user accounts. In the local development environment, PostgreSQL was set up and used locally, and the application connection was adjusted through environment variables. This allowed developers to work and test features locally before the changes were deployed to shared environments. The accounts of CMS users, such as administrators and editors, were stored in PostgreSQL. In contrast, end-user accounts on the Frontweb side were managed by an existing client-side account platform and were not included as a main component in this architecture description.
[169][Normal] The Frontweb application communicated with the CMS API to retrieve published content and configuration data. In practice, the Frontweb did not always call the CMS API directly from client-side components. To avoid CORS-related issues and to centralize data fetching logic, the Frontweb repository defined internal API routes. These internal routes then called helper functions in the Frontweb codebase, which finally fetched data from the real CMS API. This structure helped separate frontend page logic from CMS API communication and made the data-fetching process easier to maintain.
[170][Normal] Media files were stored in Cloud Storage [11]. When an administrator or editor uploaded media files such as images or videos through the CMS, the files were stored in cloud storage and could later be used by the Frontweb application. This allowed both CMS and Frontweb to share the same media resources. In the local development environment, however, Google Cloud Storage was disabled because of network and proxy limitations. Therefore, uploaded media files were stored locally during development.
[171][Normal] The system also included several supporting cloud services: Secret Manager, Cloud Monitoring, Cloud Logging, Cloud Scheduler, Cloud Run and SendGrid These services supported the overall operation of the system, although they were not the main focus of my internship tasks.

[180][Heading 2] Development Workflow
[181][isselectedend] The development workflow followed a sprint-based process, where tasks were estimated, selected for a sprint, and assigned to developers by the team leader. If there were review comments, the implementation was updated and pushed again until it was approved. After approval, the pull request was merged into develop.

Testing in the development environment helped verify that the implementation worked correctly outside the local setup.

When a completed feature was selected for staging or production release, a new branch was created from the main branch. The required commits were then cherry-picked from develop into this release branch. This approach allowed the team to control which features were included in a release, since not all features merged into develop were intended to be released immediately.
[186][isselectedend] After the selected changes were cherry-picked, a pull request was created into the main branch. When this pull request was merged, the CI/CD pipeline deployed the changes to the staging environment. After staging verification, the team leader merged the changes from main into the prd branch, which triggered the production deployment pipeline.

[195][Heading 2] NG Words Feature
[196][Heading 3] Requirement and Scope
[197][isselectedend] The NG Words feature was implemented to support content control in the CMS before articles are submitted for review or publication. In this project, NG words refer to restricted or inappropriate words that should be avoided in article content. Since the platform is used to manage promotional and product-related content, checking these words helps reduce the risk of publishing unsuitable content and supports the review process performed by CMS users.
[198][isselectedend] The requirement of this feature consisted of two main parts. The first part was NG Words management in the CMS. Authorized CMS users needed to be able to manage a centralized NG Words list, including viewing, updating, deleting, importing, and exporting NG words. The list had to be stored persistently in the system database so that it could be reused by other content validation functions. In addition, CSV import and export functions were required to make list management more efficient, especially when the number of NG words was large.
[199][isselectedend] The second part was NG Words checking during the article workflow. When an article is submitted for review, the system needs to check whether the article content contains any words from the NG Words list. This checking process is intended to reduce the manual workload of reviewers and help identify potential content issues earlier in the workflow. Instead of relying only on manual inspection, the system can automatically detect matched NG words and provide useful feedback before the review process.
[200][Heading 3] NG Words Collection and Data Persistence
[201][isselectedend] To manage the NG Words list in the CMS, a new Payload collection named NGWords was implemented. This collection represented the data model for storing restricted words used by the article checking process. Instead of creating a database table manually with SQL queries, the data model was defined at the application level through the Payload collection configuration.
[202][isselectedend] To illustrate the collection configuration, the simplified structure of the NGWords collection is shown below.
[203][isselectedend] The NGWords collection contained one main field named word. This field was defined as a text field and was marked as required and unique. The required constraint ensured that each NG Words record must contain a valid word value, while the unique constraint prevented duplicated words from being stored in the system. This was important because the NG Words list was used as a validation source, so duplicated data could make the checking process less clean and harder to maintain.
[204][isselectedend] A key characteristic of Payload CMS in this project was its schema-driven development workflow. After the NGWords collection was created and registered in the Payload configuration, a PostgreSQL migration file was generated using the command npx payload migrate:create. An example of the generated migration file is provided in Appendix B.
[205][isselectedend] The generated migration also reflected the constraints defined in the Payload field configuration. The required: true configuration was represented in PostgreSQL as a NOT NULL constraint on the word column. The unique: true configuration was represented as a unique index on the word column. As a result, the database schema remained consistent with the application-level collection definition.
[206][isselectedend] Another important point was that the feature logic did not directly manipulate PostgreSQL using manually written SQL queries. Database operations were performed through Payload CMS server-side methods. For example, when importing NG words from a CSV file, the system used Payload method payload.find to check existing records, payload.create to create new records, and payload.delete to delete old records when necessary. This approach kept the implementation consistent with Payload’s data model, access control, and internal mechanisms.
[207][Normal (Web)] This made the development process different from a traditional backend system where developers usually define database tables and queries directly. In this project, Payload acted as an abstraction layer between the application logic and PostgreSQL, allowing the NG Words feature to be implemented through collection configuration and Payload APIs while still storing the data persistently in a relational database.
[208][Heading 3] Access Control and CMS Management UI
[209][isselectedend] After defining the NGWords collection, Payload CMS automatically provided a basic management interface for the collection. Through this interface, CMS users could view the NG Words list in the admin panel. This reduced the need to build a complete CRUD interface manually and allowed the feature to follow the existing CMS structure.
[211][Caption] Figure 3: Default list view generated by Payload CMS for the NGWords collection
[212][isselectedend] Access control was added to ensure that the NG Words list could be managed safely. All authenticated CMS users were allowed to read the NG Words list. This allowed editors or other CMS users to refer to the restricted word list when needed. However, modification operations were restricted. Updating and deleting NG Words records were only allowed for users with administrative permission.
[213][isselectedend] The permission logic was implemented through the project’s role-based access control system. A user could manage NG Words if the user was a root administrator, had full permission, or had the specific NGWords:manage permission. In practice, this permission was configured so that only administrative users could modify the list, while other CMS users could only view it. This design prevented unauthorized users from changing the validation source used by the article checking process.
[214][isselectedend] In the collection configuration, the default create operation was disabled. This decision was related to the requirement that NG Words should be added mainly through the CSV import function instead of being manually created one by one through the default CMS form. This helped keep list management more controlled and suitable for bulk operations.
[215][isselectedend] To support CSV import and export, a custom toolbar component was added to the NG Words list page. This toolbar was inserted into the collection list view by using Payload’s admin component customization mechanism. The toolbar appeared before the list table and provided the actions needed to import a CSV file, export the current NG Words list, and choose the import mode.

[218][isselectedend] The toolbar also applied permission checking on the client side. If the current user did not have permission to manage NG Words, the toolbar was hidden from the interface. However, this UI-level restriction was not the only protection. The import and export API routes also checked the user’s authentication and management permission before performing any operation. Therefore, the feature used both UI-level control and server-side permission validation.
[219][Normal (Web)] This design allowed the system to reuse Payload’s generated collection interface while extending it with project-specific actions. The standard list interface was used for viewing and managing records, while the custom toolbar provided bulk import and export functionality required by the business requirement.
[220][Heading 3] CSV Import and Export Logic
[221][isselectedend] Since the NG Words list may contain many entries, managing the list manually through individual operations would be inefficient. Therefore, the system supports bulk import and full-list export through CSV files. The CSV format is intentionally simple: each line contains one NG word, and no header row is required.
[222][isselectedend] The import function supports two modes: Create new list and Insert to current list. Although these two modes differ in how they update the stored NG Words list, they share several common processing steps. In both cases, the client first validates that the selected file is a CSV file before submitting it. After the file is sent to the server, the system authenticates the user, checks the NGWords:manage permission, validates the uploaded file, reads the CSV content, splits it line by line, normalizes each word, and ignores blank lines. The normalization step helps ensure consistent storage and comparison by applying text normalization, converting words to lowercase, and trimming surrounding spaces.
[223][isselectedend] After parsing the file, the system checks whether duplicate words exist inside the uploaded CSV file. If duplicates are detected, the import process is rejected before any database update is performed. The user is then asked to correct the file before importing it again. In addition, the user can download an annotated CSV file that shows the duplicated words and their corresponding line numbers.
[225][Caption] Figure 5: Example of annotated CSV file returned
[226][isselectedend] The difference between the two import modes begins after the duplicate check. In Create new list mode, the uploaded CSV file is used to replace the entire current NG Words list. Because this action removes the previous list, the client shows a confirmation modal before the request is actually submitted. After confirmation, the server deletes all existing NG Words records and then sequentially inserts the normalized words from the uploaded file. According to the business requirement, this mode does not roll back the whole operation if some words fail during insertion. Instead, successfully imported words are kept, while failed ones are recorded and returned in the response.
[227][isselectedend] In Insert to current list mode, the existing NG Words list is preserved. After the CSV content has been parsed and validated, the system loads the current NG Words records from the database and compares them with the uploaded words. Words that already exist in the current list are skipped and recorded as failed items. Only non-existing words are inserted. This mode therefore extends the current list instead of replacing it.
[229][Caption] Figure 6: CSV import workflows for NG words management
[230][isselectedend] Figure 6 illustrates the CSV import workflows of the NG Words feature. Figure 6(a) presents the Insert to current list mode, while Figure 6(b) presents the Create new list mode.
[231][isselectedend] The export function allows authorized users to download the entire NG Words list as a CSV file. The system retrieves all NG Words records through Payload and returns a CSV file in which each line contains one word. Since the business requirement only needs full-list export, the export process does not depend on the current search or filter state of the CMS list view.
[232][Normal (Web)] If the import succeeds completely, the system returns a success response and the client shows a success message before reloading the list. If some words cannot be imported, the system returns a partial result including the number of created items and the failed words. In this case, the client shows a result modal and allows the user to download an annotated CSV file. The annotated file contains the original lines together with error notes, making it easier for the user to identify and correct problematic entries (the annotated CSV file is similar with figure 5).
[233][Heading 3] NG Words Checking Logic and UI Feedback
[234][isselectedend] The NG Words checking feature was implemented as a reusable helper function used during the article review workflow. In the submit-for-review process, this helper is called after the article passes general validation and before the article workflow status is changed to ready for review. Therefore, the main responsibility of this function is to detect restricted words in the current article content and return structured data that can be displayed in the CMS interface.
[235][isselectedend] The checking process consists of four main steps. First, the article content is extracted into plain text segments. Second, the NG Words list is loaded from the database through Payload CMS. Third, each NG word is compared with the extracted article text. Finally, if matches are found, the system builds preview data with highlighted text so that the CMS user can easily locate the problematic content.
[236][isselectedend] The fields checked by this function are title, description, and content. The title and description fields are plain text fields. Their values are split by line breaks, cleaned by removing unnecessary spaces, and converted into text segments. Empty segments are ignored.
[237][isselectedend] The content field requires a more complex extraction process because it is stored as rich text data. The helper traverses the Lexical editor structure and extracts readable text from different node types. For normal rich text nodes, paragraph, heading, and quote nodes are treated as text blocks. For list nodes, each list item is extracted as a separate segment, with a prefix representing the list type, such as bullet, numbered, or checklist item. For table nodes, each row is converted into a segment by joining the extracted text from its cells.
[238][isselectedend] The extraction logic also supports custom blocks used in the project. For a Caption Box block, the system extracts both the label and the nested content. For a CTA block, the button text is extracted and checked. For a Link Card block, only external link information is checked, including the external title and external description. For a Message Bubble node, the system extracts the speaker name and the message content. This makes the validation more complete because NG words may appear not only in normal paragraphs, but also inside structured rich text components.

[241][Normal (Web)] In this implementation, the database is used as the source of the NG Words list. The helper does not send a database query for each article segment. Instead, it retrieves NG Words records from the NGWords collection using payload.find. The records are loaded with pagination, using a limit of 1000 records per page. If the collection contains more than 1000 records, the helper continues loading the next page until all NG Words have been processed.
[242][Normal (Web)] After a page of NG Words is loaded, the helper iterates through each NG word. For each word, it checks whether that word appears in any segment of each article field. This means that each NG word is compared against the extracted text segments of title, description, and content. If a word is found in at least one segment of a field, the word is added to the result group of that field.
[243][Normal (Web)] The matching itself is performed in the application logic, not directly in SQL. This approach keeps the implementation flexible because the article content is not stored as a single simple text column. Instead, the content may contain nested rich text nodes, tables, and custom blocks. By extracting the article into text segments first and then performing the comparison in the helper function, the system can apply the same checking rule to different content structures.
[244][Normal (Web)] Before comparison, both the article text and the NG word are normalized. The normalization process applies Unicode NFKC normalization, converts text to lowercase, and removes unnecessary spaces. This reduces the risk of missing a match because of differences in character width, letter case, or surrounding spaces.
[245][Normal (Web)] The matching rule depends on the type of NG word. For Latin-based words, the system uses word-boundary matching with a regular expression. This avoids matching a word when it only appears as part of a longer unrelated word. For example, a short Latin NG word should not be detected inside another longer word if it is not a separate word. For non-Latin words, the system uses substring matching because word boundary rules are not always suitable for languages such as Japanese.
[246][Normal (Web)] When matched words are found, the helper does not only return the word list. It also builds preview data for the UI. The result is grouped by article field in an object called matchesByField. Each field group contains two main parts: words, which stores the matched NG words, and blocks, which stores preview blocks used for display.
[247][Normal (Web)] For normal text, the preview block is represented as a paragraph. Each paragraph preview contains smaller parts, and each part has a highlight value. If highlight is true, the UI displays that text as a matched NG word. If it is false, the UI displays it as normal surrounding context. To keep the preview readable, the helper does not always return the full original text. Instead, it shows the matched word with a limited number of surrounding characters and adds ellipses when the original text is longer.
[248][Normal (Web)] For table content, the preview data is represented differently. The helper identifies rows that contain matched NG words and returns the row index together with highlighted cell content. This allows the CMS interface to show which table row contains the restricted word and highlight the matched cell text.
[249][Normal (Web)] For custom blocks, the preview includes additional labels so that the user can understand where the match appears. For example, if an NG word appears in a CTA block, the preview indicates that the match belongs to the CTA button text. If it appears in a Link Card, Caption Box, or Message Bubble, the preview includes the corresponding block label and field label. This makes the result easier to understand than returning only plain text.
[250][Normal (Web)] If the helper returns a non-empty matchesByField object, the submit-for-review API returns an error response with the code ng_words_found. In this case, the article is not submitted for review and the workflow status remains unchanged. On the frontend, the returned matchesByField data is stored in the NG Words review provider and displayed in the NG Words review box in the CMS sidebar. The UI shows the affected fields, the matched NG words, and the highlighted preview content so that the author can identify and correct the problem.

[254][Heading 3] Result and Discussion
[255][isselectedend] The NG Words feature was successfully implemented as part of the CMS content management and article review workflow. The feature provides two main capabilities: managing the NG Words list in the CMS and automatically checking article content before it is submitted for review.
[256][isselectedend] For the management part, a new NGWords collection was added to Payload CMS. This collection stores restricted words in the database and allows authorized users to manage the list. The CMS interface supports viewing, updating, deleting, importing, and exporting NG Words. The import function supports two modes: creating a new list and inserting words into the current list. This provides flexibility for administrators when maintaining the restricted word list. The export function allows the full list to be downloaded as a CSV file, which is useful for backup, review, or external editing.
[257][isselectedend] The access control design also worked as expected. General authenticated CMS users can view the NG Words list, while management actions such as updating, deleting, importing, and exporting are restricted to users with the required permission. This prevents unauthorized users from modifying the validation source used in the review workflow.
[258][isselectedend] For the article review workflow, the NG Words checking logic was integrated before an article can be submitted for review. When an author clicks the Submit for Review button, the system first validates the article data and then checks the content against the NG Words list. If restricted words are detected, the submission is blocked and the article remains in its current workflow status. If no restricted words are found, the article can continue to the ready-for-review stage.
[259][isselectedend] The checking function supports multiple content structures. It checks plain text fields such as title and description, as well as rich text content stored in the article body. It can extract and check text from paragraphs, headings, quotes, lists, tables, and several custom blocks such as CTA, Link Card, Caption Box, and Message Bubble. This makes the validation more complete because restricted words may appear in different parts of an article, not only in normal paragraphs.
[260][isselectedend] Another important result is the UI feedback provided to CMS users. When NG words are found, the system returns structured data grouped by article field. The CMS interface displays the affected fields, the matched NG words, and preview content with highlighted text. This helps authors quickly identify where the problematic words are located and correct the article before submitting it again.
[261][isselectedend] Although the feature works correctly for the current requirement, there are still some limitations. First, the current matching logic loads NG Words from the database and compares each word with the extracted article text segments in application logic. This approach is simple and suitable for the expected scale of the system because the checking process runs only when an article is submitted for review. However, the processing cost increases when the article is very long or when the NG Words list becomes very large.
[262][isselectedend] Second, the checking logic needs to be maintained when new rich text blocks are added to the article editor. If a new custom block contains user-visible text, the extraction function must be updated so that NG Words can also be checked inside that block. Otherwise, restricted words inside newly introduced structures may be missed.
[263][isselectedend] Third, the matching rules may still have limitations depending on language and content format. The implementation uses whole-word matching for Latin-based words and substring matching for non-Latin words. This design reduces some false positives for English-like text and supports Japanese-like text better, but it may not cover all language-specific edge cases perfectly.
[264][isselectedend] For future improvement, the matching process could be optimized by normalizing article text segments only once before comparison. If the NG Words list grows significantly, a more efficient multi-pattern matching algorithm such as Aho-Corasick [12] could also be considered. This would allow multiple NG words to be detected in a single scan of the article text.
